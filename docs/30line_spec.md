## 0. 用語

| 語 | 意味 |
|---|---|
| 標準モード | 通常の 25/20 行。オリジナル BIOS そのまま |
| 拡張モード | 30行BIOS が INT 18H をフックし 30 行(可変 2-50)環境を作った状態。テキスト VRAM を拡張しテキストを CRT 上方へ移動 |
| Special モード | 拡張モードの G-VRAM 扱い: G-VRAM は画面中央のまま、テキストだけ移動 (テキスト/グラフィックの位置がずれる) |
| VGA モード | 拡張モードの G-VRAM 扱い: G-VRAM を T-VRAM と連動。400LINE を超える部分は G-VRAM 上部を表示 (テキスト+グラフィック重ね描き向け) |
| CW 行 | 最下行を CW(辞書変換窓?)用に確保する状態。テキスト VRAM が不連続になり最下行にオフセット 0x1F40 (51 行目相当) が表示される |

## 1. 目的と方針

VZ Editor のように 30 行テキストを使うソフト、および Canvas-98 のような 30行計画世代のソフトを
QuuBee で 30 行動作させる。
（**【訂正 2026-06-28】Canvas-98 を 30行計画世代として例示したのは誤解。恋塚氏本人より「別のアプリの話」
との指摘を受けた。詳細は §4 冒頭。**）

**鍵となる発想 (ユーザー提案):** 30BIOS.COM 本体を常駐させる必要はない。
30BIOS が常駐後にシステムへ残す**最終状態**を QuuBee の HLE が最初から用意しておけば、
アプリから見れば「30BIOS が常駐している環境」と区別がつかない。

- 30BIOS.COM / 30行計画(EE.COM 等) / 30BIOS 本体は **実 NEC CRT-BIOS ROM をパッチして常駐する**方式
  なので、ROM を積まない QuuBee では**常駐不能** (30BIOS.COM は起動直後「対応していないDOS」で自滅。
  実態は CRT-BIOS コード探索失敗 = neccheck 壁と同クラス)。→ 走らせて観測する道は無い。
- よって **静的トレース + 仕様書 (本書) で最終状態を確定 → QuuBee の合成 BIOS / HLE 層に「常駐済み 30BIOS
  相当」として実装** する。実 ROM は不要・不使用 (プロジェクトのクリーン路線を堅持)。
- クリーンな参照元: `30TECH.DOC` (公式 API 仕様)、NP2kai `bios/bios18.c` (PC-98 CRT-BIOS のオープン再実装)。
  実 ROM の入手・解析は**行わない**。

## 2. 再現すべき「30BIOS 常駐済み」最終状態

### 2.1 DOS ワークエリア (`0060:01xx` = linear `0x71x`)

30行計画世代を含むアプリの標準的な行数取得先。30行BIOS はモード移行のたびにここを更新する。

| アドレス | 内容 | 30 行(行間なし)時の値 |
|---|---|---|
| `0060:0111` (0x711) | ファンクション表示状態 (0:なし / 1:表 / 2:裏) | 現状の fkey 状態 (既存 tty が管理) |
| `0060:0112` (0x712) | 現在の最大行数 − 1 (※0x711=0 のとき実効は +1) | **29** (= 30−1) |
| `0060:0113` (0x713) | 行間状態 (0:20LINE 行間あり / 1:25LINE 行間なし) | **1** (行間なし) |
| `0060:054D` (0x54D) bit2 | GDC クロック (1:5MHz / 0:2.5MHz) | 参照のみ (FF06 で使用) |

アプリの最大行数取得の定石 (30TECH.DOC):
```asm
mov ax,0060H / mov ds,ax
mov al,ds:[0112h]            ; 最大行数 - 1
cmp byte ptr ds:[0111h],1
sbb al,-1                    ; fkey 非表示なら +1
```

> 注: 我々の `dos_int21.c` の tty は既に 0x711/0x712/0x713 を管理している
> ([[reference_pc98_doscon_workarea_0712]])。30 行時は 0x712=29, 0x713=1 を入れるよう拡張するだけ。

### 2.2 拡張 CRT-BIOS (INT 18H) — 30BIOS-API

30BIOS 固有ファンクションは **BX=0xC0A3 ('30'+'行')** を識別子に使う (これ以外の BX では既存 BIOS として動作)。
全関数、使用前に**インストールチェック必須**。

#### AH=0BH — CRT モード取得・インストールチェック
- IN : AH=0BH, BX=0xC0A3, (任意) ES:DI=`'30BIOS_EXIST=0'` を指すポインタ
- OUT: AL = 画面モード
  - bit7: 1=VGA / 0=Special
  - **bit6: 1=30BIOS 常駐 / 0=非常駐** ← 最重要のインストールチェック
  - bit5: 1=CW 行確保 / 0=fkey 表示行確保
  - bit4: 1=拡張モード / 0=標準モード
  - bit3〜1: オリジナル BIOS と同じ
  - bit0: 1=行間あり(拡張時)/20行(標準時) / 0=行間なし(拡張時)/25行(標準時)
  - ES:DI が `'30BIOS_EXIST=0'` を指していたら `'...=1'` に**書き換えて返す** (Ver0.20+ の厳密チェック)
- 注: オリジナル BIOS の「CRT モードのセンス」では bit6 は常に 0。
  我々の現状: `bios0x18` の case 0x0B は `AL=mem[0x53C]` を返すだけ → bit6 を立てる拡張が要る。

#### AH=0AH (BX=0xC0A3) — CRT モード移行
- IN : AH=0AH, BX=0xC0A3, AL=(bit7 VGA/Special, bit5 CW/fkey, bit4 拡張/標準, bit0 行間)
- 動作: GDC パラメータ再設定 + DOS ワークエリア再設定 (+実機では IO.SYS の 20↔25 切替コードにパッチ)
- 注: BX≠0xC0A3 の AH=0AH は**オリジナル拡張**版 (下記 2.4) として動く

#### AX=FF00H — バージョン取得
- OUT: AH=小数部, AL=整数部 (例 Ver1.10 → AH=10h?/AL=01h。値は名乗り次第。提案: AL=1, AH=20h 相当)

#### AX=FF01H / FF02H — 画面モードの PUSH / POP (8 段スタック)
- OUT: AX=FFFFH 成功 / 0000H 失敗。保存データ = AH=0BH の AL + 現在行数 の 2 バイト

#### AX=FF03H — 画面行数の変更・取得
- IN : AX=FF03H, BL=行間なし時の行数 (BL=00 で取得のみ)
- OUT: AL=行間なし時の行数−1, AH=行間あり時の行数−1
- 注: 変更後は即 AH=0AH でモード移行が必要。`(AL+1)≠BL` なら上限/下限クランプ (CRT 保護)。
  BL=00 取得時は**現在のモードに関係なく拡張モード時の行数** (= 標準モードでも 30 が返る)

#### AX=FF04H — 設定可能行数の取得 (Ver0.20+)
- OUT: AL=上限, AH=下限

#### AX=FF05H — 行間空き時のラスタ数 変更/取得 (Ver1.10+)
- IN : BH=ラスタ数, BL=10H 変更 / 00H 取得 / OUT: AH=行間ありラスタ, AL=行間なしラスタ(=10H)

#### AX=FF06H — GDC パラメータ取得 (Ver1.10+)
- IN : BL=行間なし時の行数 / OUT: AL=HS+1, AH=VS, CL=VBP, CH=VFP, DL=HBP+1, DH=HFP+1 (5MHz 時)
- 2.5MHz 時は AL,DL,DH を 1/2 して −1。SI,DI 破壊・BX 保存

### 2.3 フックされる既存ファンクション (BX≠0xC0A3)

30行BIOS は以下を**フックして拡張モード時の整合**を取る (新規ではなく既存 BIOS と同じ顔)。

- **AH=0AH (BX=0)**: CRT モード設定 (オリジナル拡張)。20↔25 行移行を拡張モードでも画面を崩さず処理。
  bit0=行間隔。
- **AH=0EH**: 1 つの表示領域設定。拡張モード時はテキスト VRAM オフセットを補正してオリジナルを呼ぶ。
- **AH=0FH**: 複数表示領域設定。DL=04 or 標準モード時は素通し。
- **AH=42H**: 表示領域設定。GDC 5MHz で 200→400LINE 変更時のグラフィック崩れ防止。標準時は素通し。

> QuuBee では CLS/カーソル/スクロールを自前 tty (`dos_int21.c`) が処理しているため、実機 30BIOS の
> IO.SYS パッチ (画面分割テーブル / X,Y→VRAM 変換 / CLS 行数 / CTRL+f10) を**そのまま再現する必要はない**。
> 我々の tty を 30 行対応にすれば等価。

### 2.4 GDC / 画面

- 拡張モードは **30 行テキスト = 480px (30×16)** を表示する。エミュレータでは EXTs21 で実証済の 480 ライン
  表示経路 (`bios0x18_30` / `vram/dispsync.c` → `scrnmng_setheight(0,480)`) がそのまま使える
  (`SUPPORT_CRT31KHZ`+`SUPPORT_PC9821` 有効、fb は 640×480 へ自動 realloc)。フォントセルは 16px 据置。
- グラフィックの Special / VGA は **Canvas のような重ね描き/拡張キャンバスでのみ問題**になる。テキスト 30 行
  だけ (VZ 系) なら不要。詳細は §4。

## 3. QuuBee 実装マッピング

### 3.1 INT 18H フロントエンド

現状 INT 18H は NP2kai の `bios0x18` (合成 BIOS) が処理。30BIOS-API を足す手段は 2 案:

- **案 A (推奨): 自前トランポリン + パススルー**
  `0xFEEB0` 等に新規トランポリン (NOP+IRET)。IVT[0x18] を我々のハンドラへ向け、C 側 `qb_dos_int18_hook` が
  - BX=0xC0A3 / AX=FFxx → 30BIOS-API として処理
  - それ以外 → `bios0x18()` を直接呼んでパススルー
  既存の NOP-hook 機構 ([[feedback_np2kai_nop_hook]] / [[feedback_dos_loader_trampoline]]) と同型。
  **core 改変ゼロ**を保てる (Canvas/PMD と同じ流儀)。
- 案 B: `core/np2kai/bios/bios18.c` の case 0x0A/0x0B に 30BIOS 分岐を直接追加 + FFxx を足す。
  実装は素直だが core 改変になる。

> どちらでも、AH=0BH の bit6 立てと ES:DI 文字列書換、AX=FF03H の行数取得が最低限。

### 3.2 GDC 30 行表示

- 拡張モード ON 時に 480 ライン表示へ移行 (`bios0x18_30` 経由か、GDC へ直接)。`scrnmng` が 480 へ追従。
- テキスト VRAM 30 行 (80×30×2=4800B、テキストプレーン 8KB 内に収まる)。

### 3.3 tty / ワークエリア (`dos_int21.c`)

- `TEXT_ROWS` を 25 固定から **可変 (25/30)** へ。カーソル/スクロール/CLS が新行数で動くように。
- `qb_dos_tty_reset` の 0x711/0x712/0x713 セットを 30 行時 (0x712=29, 0x713=1) に対応。

### 3.4 トリガ (どう 30 行を ON にするか)

標準 PC-98 にはコンソールで 30 行にする手段が無い (前調査の結論)。よって QuuBee 独自トグル:

- `np2kai_set_lines30(int on)` (bridge.h、既存 `np2kai_set_chibioto` 等と同パターン) +
  `qbDebug.lines30(1)` (JS)。**既定 OFF** (画面既定を変えない)。
- ON 時、次 reset で「30BIOS 常駐済み」最終状態 (§2) を構築。

## 4. Canvas-98 (Phase 2) — 調査結論: いったん見送り (2026-06-28)

> **【訂正 2026-06-28】** 本節は当初 Canvas-98 (canvas.exe 1992/12/30、©恋塚) を「30行計画世代のソフト」と
> みなし 30 行表示の検証対象に据えていたが、**恋塚氏本人より「それは別のアプリの話」との指摘を受けた**。
> Canvas-98 を 30行BIOS / 30行計画 の文脈に結びつけた理解は誤解を含む。本節の技術的調査 (canvas.exe は
> 30BIOS-API を使わず行数ワークエリアも読まない = 環境を検出しない) は記録として残すが、これはむしろ
> 「Canvas-98 は 30 行表示とは別系統」という恋塚氏の指摘と整合する。下記「再挑戦するなら恋塚氏に確認」は
> 取り下げる。なお下記の VRAM 物理制約の結論はアプリに依らず成立する。

実機テスト 2 回 + canvas.exe 静的解析で素性が判明したが、**現状の HLE では正しく出せず見送り**とした。

判明した素性:
- canvas.exe (1992/12/30) は **30BIOS (1993/4) より前**で 30BIOS-API を使わない (INT 18H は AH=40/41/42 のみ)。
- **画面高さ `maxY = [DS:0x7a] × 16`、かつ `[0x7a]` はどこからも書かれない＝静的定数** (実質ハードコード。
  "C30" = その値を変えたビルド)。つまり Canvas は環境を**検出せず**、決め打ちのレイアウトを描く。
- 起動時に自分で INT 18H AH=42h CH=0xC0 を呼び 640×400 に戻す。

実機の症状 (lines30 ON + AH=42h 横取りで 480 維持を試した版): 画面は 480 に伸びるが、ツールバーが
中途半端な位置・**下 80 ラインが画面上部の繰り返し・カーソル 2 個**。

根本原因 = **PC-98 の 16 色グラフィック VRAM は 1 プレーン 0x8000B = 640×400 ちょうどで、480 ライン分が物理的に
入らない**。480 表示にすると下 80 ラインは VRAM 先頭 (画面上部) を繰り返す。これは 30行計画/30行BIOS でも同じで、
30TECH.DOC も拡張モードのグラフィックは Special/VGA とも「400 ラインを超える部分は G-VRAM 上部を表示 (繰り返し)」
と明記。**30 行で増えるのはテキスト面の行数**で、グラフィックは 400 ライン据置。Canvas はこの前提で「絵=グラフィック
400 ライン / ツールバー等=テキスト面の拡張行」をプレーン別に重ねるが (DOC「座標表示にテキスト画面を使用」「テキストと
グラフィックの位置関係がずれる」)、この**テキスト面 (30 行) × グラフィック面 (400+繰り返し) の重ね合わせを Canvas が
期待する正確な形に再現できていない**ため崩れる。

→ Canvas を動かすには、Canvas のプレーン別レイアウト (どの行に何を、テキスト/グラフィックどちらに) を精密に合わせる
深い作業が要る。テキスト系 (VZ) と別クラスの難物なので **Phase 1 を実用到達点として一区切り**とした。

再挑戦するなら: 恋塚氏に「30 行時、Canvas はツールバー/絵をテキスト面・グラフィック面のどちらの何ラインに置くか/
グラフィックのページ切替の具体」を確認 → 実現可否を判断。試した AH=42h 横取り (480 維持) は revert 済
(Canvas を直せず・他アプリへの影響回避のため。再挑戦時の出発点として本節に記録)。

> 補足: maxY=480 を前提に描く Canvas で「下 80 が上の繰り返し」になる以上、16 色 480 ライン linear が要る。
> PC-9821 の native 480 は **256 色 (PEGC)** 経路で、Canvas (16 色描画) はそれを使わない。よって 256 色化でも
> 解決しない。Canvas 流の「テキスト面に道具・グラフィック 400 に絵」を忠実に再現する以外に筋が見えていない。
